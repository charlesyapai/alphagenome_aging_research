# v1 → v2 postmortem

This is a short accounting of what went wrong in v1 (`Idea Testing 1/`, originally labeled "AgingAtlas") and how v2 fixes each issue. It exists so a reviewer landing on the v2 code can see the methodology audit trail.

## v1 headline claim

> "A Random Forest classifier distinguishes aging from control variants with **92.1% accuracy** (AUC 0.863, permutation p < 0.001, 200 shuffles)."  
> — `Idea Testing 1/docs/SESSION_CATCHUP.md`, line 14

This was the number that would have gone on a portfolio card.

## What was actually happening

| | Observation | Impact |
|-|---|---|
| **Class imbalance** | 5,211 aging vs 500 controls. Majority-class baseline: 91.24%. | The "92.1%" accuracy is +0.9 points above a trivial classifier. |
| **Controls not MAF-matched** | Planning doc specified MAF + gene-distance matching. Code implemented chromosome-matching only. 93% of controls had unknown MAF in `control_maf_info.csv`. | Aging variants are preferentially common (mean AF ≈ 0.31); controls skewed rare. Classifier could be learning "common vs rare", not "aging vs not-aging". |
| **LD leakage via naive CV** | `StratifiedKFold` shuffles rows independently. Many aging variants cluster in the same gene (APOE, TERT, FOXO3). | Correlated variants on both sides of train/test boundary inflate CV metrics. |
| **Preprocessing leakage** | `StandardScaler.fit_transform` ran on combined aging+control data in `run_expanded_analysis.py:411-413` before CV. | Small leakage for RF (scale-invariant), but a real leak for any linear model and a bad smell for methodology. |
| **No held-out test set** | All metrics were CV-based. | No truly unseen data; no check that the model generalizes. |
| **Sensitivity analysis had same bug** | Running on GWAS-significant subset of aging (n=4,983) vs same 500 controls → 91.8% accuracy. | The "sensitivity" was still dominated by class imbalance, not sensitivity of the result to variant selection. |

## v2 fixes, one-to-one

| v1 problem | v2 fix |
|---|---|
| Class imbalance inflating accuracy | 1:1 balanced design (~5k aging, ~5k MAF-matched controls). Accuracy and balanced accuracy agree. |
| Non-MAF-matched controls | For each aging variant, MAF-bin-matched control from same chromosome neighborhood, drawn from the 1000 Genomes GRCh38 high-coverage phased panel via remote tabix fetch. Smoke test: 100% match rate, per-bin matched distribution mirrors aging bin distribution exactly. |
| LD leakage via naive CV | `GroupKFold` by gene. Gene assignment from VEP, falling back to nearest-gene, falling back to 100-kb buckets. |
| Preprocessing leakage | `SimpleImputer` + `StandardScaler` inside `sklearn.Pipeline`, fit per training fold inside CV. |
| No held-out test set | 20% of gene groups held out via `GroupShuffleSplit`. All final metrics reported on this set. |
| Raw accuracy as headline | Balanced accuracy, AUC, PR-AUC, Brier reported together. Raw accuracy visible but never alone. |
| No leakage audit | Grouped CV AUC and naive CV AUC both reported; the delta is the leakage v1 had. |
| No regression guard | `tests/test_no_leakage.py` — pytest suite that asserts each methodology invariant. |

## v2.0 → v2.1 evolution

v2 shipped in two rounds because the first attempt to collect MAF-matched controls hit an external dependency wall:

- **v2.0** (first working build): retained v1's 500 chromosome-matched controls. Training was honest (gene-grouped CV, held-out test, calibration reported), but the 10:1 class imbalance meant balanced accuracy (0.66) rather than raw accuracy (0.91) was the informative number. Committed with metrics.json and full docs; metrics snapshotted to `results/snapshots/metrics_v2.0_*.json`.
- **v2.1** (current): swapped Ensembl REST — which was timing out and returning 0% match rate at scale — for direct remote tabix queries against the 1000 Genomes GRCh38 high-coverage phased panel via `pysam.VariantFile`. 200× faster, 100% match rate, MAF pulled directly from the VCF `INFO/AF` field. Class ratio moves from 10:1 to 1:1.

The `results/RUN_COMPARISON.md` file (generated by `scripts/compare_runs.py`) shows v2.0 vs v2.1 side-by-side. The point of keeping both is that a reviewer can see the magnitude of the class-imbalance fix independent of anything else.

## What we did NOT change

- **AlphaGenome scoring.** v1's 5,404 score files are copied into v2 verbatim. Re-scoring would waste API calls and change nothing.
- **Aging variant list.** The 5,211 curated variants stay. The question we're asking is the same; only the comparison pool changes.
- **Feature aggregation.** 115 summary features — same shape, same logic, same code path between train and serve.
- **Random Forest as default classifier.** No claim that RF is best; it's a reasonable default and swappable via `make_*_pipeline` factories.

## What remains honest-but-limited

- **Controls are frequency-matched, not trait-negative-verified.** A gnomAD common variant nearby an aging variant is unlikely to be an aging hit, but we do not verify that.
- **Heterogeneous "aging" label.** Six sources; some effects are cardiovascular disease mechanisms, some are longevity per se, some are telomere biology. The multi-class trait task partially addresses this but labels are still coarse.
- **~10k total variants is small.** Any trained classifier on this scale is a useful prior, not a diagnostic.

## Why this matters for a portfolio

The 92.1% → ~(real balanced) number change is the point. A reviewer who glances at v1 sees a flashy metric; a reviewer who reads v2 sees a project that interrogated its own numbers, caught the inflation, and rebuilt around an honest design. The honest number is the better portfolio signal.
